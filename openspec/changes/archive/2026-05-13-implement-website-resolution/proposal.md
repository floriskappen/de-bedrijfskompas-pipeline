# Proposal: Implement the website resolution stage

## Why

The pipeline needs a canonical website URL for every company before any downstream stage can run. Future inputs will arrive with only a company name, so the first stage must *discover* a website when missing. The MVP source is `test-set/companies.json` (which already includes websites — used for verifying the discovery step finds them correctly).

## What Changes

- Introduce the `website-resolution` stage — stage 1 per `pipeline-architecture`.
- Input shape: a JSON record with at least `name` (required) and optionally `website` (and arbitrary other keys carried through).
- Behavior:
  - If `website` is already present in the input, the stage skips discovery for that record and passes the record through unchanged.
  - If `website` is missing, use the `DDGS` Python library to issue a search-engine query for the company name, region=NL, and take the top result as the resolved website.
- Output shape: same record as input with `website` added/populated.
- Failure handling: when no usable result is found (no hits, DNS/HTTP errors, etc.), write an error record `{...input, website: null, status: "failed", error: ...}` and continue with the next company.
- Test approach: at test time, strip `website` from the three entries in `test-set/companies.json`, run the stage, and assert each recovered URL matches the original.
- **Modify `pipeline-architecture`** to add a "stage execution model" requirement: every stage MUST be (a) runnable standalone as a module, (b) callable by a future orchestrator, and (c) supportable in a no-write/dry-run mode so tests can run without producing files on disk.

## Capabilities

### New Capabilities
- `website-resolution`: takes a company record (name required, website optional), produces the same record with a canonical `website` field, or an error record on failure. Uses `DDGS` for search.

### Modified Capabilities
- `pipeline-architecture`: add a new requirement covering the stage execution model (standalone CLI + orchestrator-callable + dry-run/no-write mode). Does not change existing requirements.

## Impact

- **Code:** First production code in the repo. A new module under `pipeline/website_resolution/` with a CLI entry point.
- **Dependencies:** `DDGS` (search), `httpx` or similar (if any HTTP verification is added later). Experimentation needed on which DDGS engine performs best for NL companies and how to handle rate limits without proxies.
- **Data:** Creates `data/website-resolution/` populated from a run over `test-set/companies.json` (or any other minimal `{name}` input list).
- **Open question:** disambiguation when multiple companies share a name. Not resolved here — to be addressed in design.md.
